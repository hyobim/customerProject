# 구현 결정 기록

## 메모리 저장소

전화번호를 키로 하는 `HashMap<String, Customer>`와 이메일에서 전화번호를 찾는 보조 `HashMap<String, String>`을 사용했다. 정확 검색은 평균 O(1)이고 별도 DB 없이 이메일 유일성도 빠르게 검사할 수 있다.

두 Map의 복합 변경은 `ReentrantReadWriteLock`으로 보호한다. 단일 `ConcurrentHashMap`만으로는 전화번호 Map과 이메일 인덱스를 함께 바꾸는 작업의 원자성을 보장할 수 없다. 고객은 불변 record로 만들어 잠금 밖에서 내부 값이 변경되지 않게 했다.

이름과 주소 부분 검색은 스냅샷을 만든 뒤 O(n) 순회한다. 현재 과제 규모에서 n-gram 인덱스의 메모리와 갱신 복잡도보다 단순 순회가 적합하다.

## 식별자 변경

실무에서는 변경 가능한 전화번호 대신 불변 고객 ID를 기본 키로 두는 편이 일반적이다. 이 과제에는 별도 ID가 없고 전화번호가 키로 지정되어 있으므로 경로의 기존 전화번호로 대상을 식별하고 본문의 새 전화번호로 원자적으로 변경한다.

## 검증

HTTP 요청 DTO는 Bean Validation으로 필수값과 기본 형식을 빠르게 거부한다. 이후 CSV와 API가 같은 `CustomerValidator`를 사용해 최종 검증과 정규화를 수행한다. 전화번호는 검증 후 표준 하이픈 형식으로 정규화한다. 이메일은 요구사항에 없는 소문자 변환을 하지 않고 대소문자를 구분한다.

## CSV

외부 의존성을 늘리지 않고 필드 내 쉼표, 이중 따옴표, 줄바꿈을 지원하는 작은 스트리밍 CSV 코덱을 구현했다. 전체 파일 문자열을 메모리에 올리지 않고 레코드 단위로 읽는다.

저장은 원본 백업, 같은 디렉터리의 임시 파일 쓰기, 원자적 이동 순서다. 파일 시스템이 원자적 이동을 지원하지 않으면 교체 이동으로 대체한다. 임시 파일 쓰기가 실패하면 기존 원본은 유지된다.

## 알려진 제약

- 조회 페이지네이션이 없어 결과가 매우 크면 응답 메모리가 커질 수 있다.
- 비정상 강제 종료에서는 `@PreDestroy` 저장이 실행되지 않을 수 있다.
- 초 단위 백업 파일명은 같은 초에 여러 번 저장하면 기존 백업을 교체한다.
- 현재 작업 경로의 한글 정규화 문제로 Windows Gradle/JUnit 실행 시 영문 경로 교차 검증이 필요하다.
