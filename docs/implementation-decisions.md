# 구현 결정 기록

## 메모리 저장소

전화번호를 키로 하는 `HashMap<String, Customer>`와 이메일에서 전화번호를 찾는 보조 `HashMap<String, String>`을 사용했다. 정확 검색은 평균 O(1)이고 별도 DB 없이 이메일 유일성도 빠르게 검사할 수 있다.

두 Map의 복합 변경은 `ReentrantReadWriteLock`으로 보호한다. 단일 `ConcurrentHashMap`만으로는 전화번호 Map과 이메일 인덱스를 함께 바꾸는 작업의 원자성을 보장할 수 없다. 고객은 불변 record로 만들어 잠금 밖에서 내부 값이 변경되지 않게 했다.

이름과 주소 부분 검색은 스냅샷을 만든 뒤 O(n) 순회한다. 현재 과제 규모에서 n-gram 인덱스의 메모리와 갱신 복잡도보다 단순 순회가 적합하다.

## 식별자 변경

실무에서는 변경 가능한 전화번호 대신 불변 고객 ID를 기본 키로 두는 편이 일반적이다. 이 과제에는 별도 ID가 없고 전화번호가 키로 지정되어 있으므로 경로의 기존 전화번호로 대상을 식별하고 본문의 새 전화번호로 원자적으로 변경한다.

## 검증

HTTP 요청 DTO는 Bean Validation으로 필수값과 기본 형식을 빠르게 거부한다. 이후 CSV와 API가 같은 `CustomerValidator`를 사용해 최종 검증과 정규화를 수행한다. 전화번호는 검증 후 표준 하이픈 형식으로 정규화한다. 이메일은 요구사항에 없는 소문자 변환을 하지 않고 대소문자를 구분한다.

전화번호·이메일 정규식과 오류 메시지는 `CustomerValidator`의 상수로 공유한다. DTO와 업무 검증이 서로 다른 규칙으로 변하는 위험을 줄이되, 필수값 같은 HTTP 입력 경계 검증은 DTO에 남긴다.

Controller에서 Service로 전달하는 조회 인자는 `CustomerSearchRequest`로 묶었다. HTTP 쿼리 파라미터 계약은 유지하고, 내부 메서드의 인자 순서 실수와 확장 비용만 줄인다.

`CustomerSearchRequest`는 HTTP 전용 DTO가 아니므로 `controller.dto`에 두지 않는다. Service가 Controller 패키지에 의존하면 계층 의존 방향이 역전되기 때문이다. 반면 원시 `sortBy`, `direction` 문자열을 가진 객체는 핵심 도메인 모델도 아니므로 `domain` 대신 `service.dto`에 두어 애플리케이션 서비스 입력 DTO임을 명시한다.

현재 HTTP 검색 조건과 Service 검색 입력의 필드가 완전히 같고 별도 변환 규칙이 없으므로 Controller에서는 `@ModelAttribute CustomerSearchRequest`로 쿼리 파라미터를 직접 바인딩해 재사용한다. 두 계약이 달라지는 시점에는 HTTP DTO와 Service 입력 객체를 분리한다.

## Lombok 적용

요청·응답 DTO와 검색 입력 DTO는 같은 타입의 필드가 여러 개이므로 생성 시 필드 순서 실수를 줄이기 위해 Lombok `@Builder`를 제공한다. record에도 `@Builder`를 적용할 수 있으며, record의 불변 값 객체 특성과 Builder의 이름 기반 생성 편의는 서로 대체 관계가 아니다. Jackson은 기존 record 정규 생성자를 사용하므로 HTTP JSON 계약은 바뀌지 않는다.

단순한 `final` 의존성만 주입받는 `AddressBookController`와 `AddressBookService`는 `@RequiredArgsConstructor`로 생성자 코드를 줄였다. 설정값을 변환하거나 생성자 파라미터에 `@Value`가 필요한 `CsvCustomerStore`, `CustomerDataLifecycle`은 생성 과정 자체가 의미를 가지므로 명시적 생성자를 유지한다.

## CSV

외부 의존성을 늘리지 않고 필드 내 쉼표, 이중 따옴표, 줄바꿈을 지원하는 작은 스트리밍 CSV 코덱을 구현했다. 전체 파일 문자열을 메모리에 올리지 않고 레코드 단위로 읽는다.

UTF-8 BOM은 파일 시작의 헤더 첫 필드에서만 제거한다. 데이터 필드에 포함된 BOM 문자는 임의로 변경하지 않는다.

CSV 읽기는 필드 시작, 일반 필드, 따옴표 필드, 닫는 따옴표 직후의 네 상태를 명시적으로 구분한다. 일반 필드 중간의 따옴표와 닫는 따옴표 뒤의 일반 문자·공백은 문법 오류로 거부한다. 문법 오류는 레코드 경계를 신뢰할 수 없으므로 행 단위로 복구하지 않고 `CustomerDataFileException`을 전파해 기동을 실패시킨다. 반면 열 개수, 고객 필드 형식과 중복 오류는 레코드 경계가 유효하므로 기존처럼 해당 레코드만 건너뛴다.

저장은 원본 백업, 같은 디렉터리의 임시 파일 쓰기, 원자적 이동 순서다. 파일 시스템이 원자적 이동을 지원하지 않으면 교체 이동으로 대체한다. 임시 파일 쓰기가 실패하면 기존 원본은 유지된다. 원본 교체가 실패하면 방금 생성한 백업으로 원본 복구를 시도하고, 복구 실패는 최초 교체 예외의 suppressed exception으로 보존한다.

## 빌드

Java 버전은 Gradle Toolchain으로 17을 명시한다. 테스트 플랫폼 런처도 런타임 의존성으로 선언해 Gradle 9에서 제거 예정인 암묵적 테스트 프레임워크 로딩에 의존하지 않는다.

## 알려진 제약

- 조회 페이지네이션이 없어 결과가 매우 크면 응답 메모리가 커질 수 있다.
- 비정상 강제 종료에서는 `@PreDestroy` 저장이 실행되지 않을 수 있다.
- 초 단위 백업 파일명은 같은 초에 여러 번 저장하면 기존 백업을 교체한다.
- 페이지네이션과 백업 파일명 충돌 정책은 외부 계약 변경이므로 별도 결정이 필요하다.
