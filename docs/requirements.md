[요구사항 정의서] 고객 정보 관리 프로그램
1. 프로젝트 개요
고객 정보를 컴퓨터로 관리하고자 함. 
관리하는 고객 정보는 이름, 주소, 전화번호, 이메일 총 4가지임.
주어진 초기 코드를 바탕으로, 요구사항을 만족하는 코드를 어플리케이션이 동작하도록 구현해야함.
해당 프로젝트는 3년차 주니어 서버 개발자의 아키텍쳐/백엔드 코드 설계 능력을 판단하는 간단한 과제 프로젝트임. 이 수준에 맞는 개발이 필요하고 모든 설계에 판단 근거가 필요함. 

2. 서비스 구성 및 기술 스택
- JAVA로 구현 
- clean code, 깔끔한 아키텍쳐 사용
- 주어진 코드에 문제가 있거나 개선의 필요성이 있을 경우 docs로 명시 후 적절하게 수정
- DTO 등의 데이터 모델 객체를 자유롭게 활용
- 모든 코드는 동시성 문제와 성능을 고려하여 작성, 메모리 / 검색 성능 등을 잘 조율하여 개발 > 이부분에 대해서는 docs로 선택 기준 및 판단 공유
- 데이터의 조회 결과에 따라 발생하는 에러 및 응답 코드는 http status code 기반으로 작성

3. 핵심 기능 요구사항 (MVP 스펙)
3.1 조회
파라미터 
필터 : 전화번호, 이메일, 주소, 이름 (이름/주소 조회 시 양방향 like 검색 구현)
정렬 : (전화번호|이메일|주소|이름), (오름차순|내림차순)

응답 : 고객 정보 목록을 json형식으로 응답

3.2 수정
파라미터 : 고객정보 
응답: 고객 정보 json. 수정 전 데이터와 수정 후 데이터를 구분해서 반환.

3-3 삭제
파라미터 : 전화번호 or 이메일 or 주소 or 이름 (이름/주소 삭제 시에는 like 검색하여 삭제 처리)
응답: 삭제된 고객정보 json.

3-4 어플리케이션 중지 시 
어플리케이션 중지 시 기존 파일은 백업, 메모리에 로딩되어있는 내용으로 저장되도록 작성 

4. MVP 기술 개발 제약 명세 (Technical Specifications)
4.1 데이터 로딩 시 아래 형식/항목이 맞는지 확인하여야 함.
4.1.1 데이터 형식
- 전화번호는 010으로 시작해야함.
- 전화번호 형식은 0101231234 또는 010-123-1234 또는 010-1234-1234이어야 함. 
- 이메일 형식은 '아이디@도메인' 이어야 함.

4.1.2 데이터의 키는 전화번호이다.
4.1.3 이메일은 중복될 수 없다. 
4.1.4 메모리에 데이터를 로딩할 때 형식을 만족하지 못하는 경우, Warning 로그를 작성한 뒤 skip하고 이후 데이터를 읽어야 함.
4.1.5 사용자가 호출하는 API인 경우, 오류를 stack trace로 출력하는 것과 별개로 사용자에게 노출될 메시지를 API 응답으로 출력해야함.

4.2 테스트 케이스 필수적으로 작성
4.2.1 각 클래스의 단위 테스트를 작성
- 초기 로딩 검증
- 기본 기능 각각에 대한 검증
- 프로그램 종료 시 최종 파일 반영 프로세스 검증 

4.2.2 Controller 단위 테스트의 경우 MockMvc를 이용한 테스트로 작성
- org.springframework.test.web.servlet.MockMvc

4.2.3 긍정 테스트와 부정 테스트 각각 동일한 개수로 생성 후 진행
4.2.4 단위 테스트 이후 통합 테스트 진행
