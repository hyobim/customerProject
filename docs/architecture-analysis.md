# 기존 코드 아키텍처 및 데이터 흐름 분석

## 1. 분석 목적과 범위

이 문서는 루트의 `Readme.md`, `docs/requirements.md`, `docs/constraints.md`, 현재 Java 소스, 테스트 스켈레톤, Gradle 설정, CSV 샘플을 기준으로 프로젝트의 현재 상태와 의도를 분석한 결과다.

`Readme.md`는 최초 과제 안내와 기본·추가 요구사항을 함께 담고 있으며, `requirements.md`는 이를 세분화하면서 긍정/부정 테스트 수 균형과 단위 테스트 이후 통합 테스트 같은 검증 조건을 추가한다. `constraints.md`는 기술 선택의 금지 사항과 문서 언어를 정의한다. 세 문서는 서로 충돌하지 않으므로 합집합을 구현 기준으로 사용한다.

현재 단계에서는 개발 계획 승인 전이므로 애플리케이션 기능 구현은 수행하지 않았다. 분석과 계획 문서만 추가한다.

## 2. 프로젝트 의도

이 프로젝트는 RDBMS나 임베디드 DB 없이 고객 주소록을 관리하는 Spring Boot 기반 REST API를 구현하는 과제다. 핵심 평가 대상은 단순 CRUD 자체보다 다음 항목에 있다.

- 요구사항을 명확한 API 계약과 도메인 규칙으로 변환하는 능력
- 조회 결과와 오류 유형을 의미에 맞는 HTTP 상태 코드로 표현하는 능력
- 파일, 메모리, HTTP 계층의 책임을 분리하는 아키텍처 설계 능력
- 전화번호 기본 키와 이메일 유일성 제약을 동시성 상황에서도 일관되게 유지하는 능력
- 전체 탐색이 필요한 이름/주소 부분 검색과 키 기반 검색의 성능을 합리적으로 조율하는 능력
- 잘못된 초기 데이터를 격리하고 정상 데이터 로딩을 계속하는 복원력
- 정상 종료 시 백업과 파일 저장을 안전하게 수행하는 영속화 설계 능력
- 단위 테스트, MockMvc 테스트, 통합 테스트로 설계를 검증하는 능력
- 설계 선택과 트레이드오프를 문서로 설명하는 능력

## 3. 현재 코드 구조

현재 소스 구조는 다음과 같다.

```text
com.hyundai.test.address
├── Application
├── common
│   └── CsvFileReader
└── controller
    └── AddressBookController
```

테스트 구조는 다음과 같다.

```text
com.hyundai.test.address
├── controller
│   └── AddressBookControllerTest
└── service
    └── AddressBookServiceTest
```

### 3.1 `Application`

- Spring Boot 실행 진입점이다.
- `DataSourceAutoConfiguration`을 제외하여 DB를 사용하지 않는 제약과 방향은 일치한다.
- 현재는 초기 데이터 로딩이나 종료 저장을 연결하는 생명주기 구성이 없다.

### 3.2 `AddressBookController`

- `@RestController`만 선언된 빈 클래스다.
- URL, HTTP 메서드, 요청/응답 DTO, 서비스 의존성, 상태 코드가 전혀 정의되어 있지 않다.

### 3.3 `CsvFileReader`

- 클래스패스 리소스를 한 줄씩 읽어 `List<String>`으로 반환한다.
- 다음 문제가 있다.

1. 클래스패스 리소스는 패키징된 애플리케이션에서 일반적으로 쓰기 대상이 될 수 없으므로 종료 시 원본 백업 및 덮어쓰기 요구사항과 맞지 않는다.
2. 리소스가 없을 경우 `getResourceAsStream()`이 `null`을 반환하여 명확한 파일 예외가 아니라 `NullPointerException`이 발생할 수 있다.
3. 스트림과 리더를 닫지 않아 자원 누수 가능성이 있다.
4. 문자열 `"UTF-8"` 대신 `StandardCharsets.UTF_8`을 사용할 수 있다.
5. 모든 줄을 먼저 메모리에 적재하므로 대용량 파일에서 불필요한 중간 메모리를 사용한다.
6. CSV 구문 분석, 헤더 검증, 열 개수 검증, 따옴표와 쉼표 처리, 데이터 검증 책임이 없다.
7. 로딩 전용 추상화이므로 저장, 임시 파일, 백업, 원자적 교체를 지원하지 않는다.

현재 클래스는 일부를 수정해 재사용하기보다, 파일 영속성 책임을 명확히 가진 컴포넌트로 대체하는 편이 적절하다.

### 3.4 테스트

- 두 테스트 클래스 모두 실질적인 테스트 메서드가 없다.
- Controller 테스트는 `@SpringBootTest`, `@AutoConfigureMockMvc`만 선언되어 있다.
- 초기 로딩, 조회, 수정, 삭제, 유효성 검증, 중복 검증, 종료 저장 테스트가 모두 없다.
- 단위 테스트와 통합 테스트의 경계가 정의되어 있지 않다.

## 4. 현재 데이터 흐름

현재 구현된 데이터 흐름은 없다.

```text
HTTP 요청 ─X─> Controller ─X─> Service ─X─> 메모리 저장소
CSV 파일 ─X─> 검증/파싱 ─X─> 메모리 저장소
메모리 저장소 ─X─> 백업/CSV 저장
```

`CsvFileReader`를 직접 호출하면 클래스패스 파일의 원문 문자열 목록을 얻을 수 있을 뿐이며, 이 결과는 도메인 객체나 API와 연결되어 있지 않다.

## 5. 데이터 파일 분석

### 5.1 `default_address.csv`

프로젝트 루트에 있으며 요구사항상 실제 읽기/쓰기 대상일 가능성이 가장 높다.

- 헤더: 주소, 연락처, 이메일, 이름
- 데이터 행: 7개
- 요구사항을 문자 그대로 적용할 때 예상되는 처리:
  - 정상 행은 메모리에 적재한다.
  - 동일 이메일 `lee2@hyundai.com`의 후속 행은 중복으로 건너뛴다.
  - 이메일 `lee3` 행은 형식 오류로 건너뛴다.
  - 마지막 행의 전화번호는 앞선 잘못된 이메일 행이 적재되지 않았다면 전화번호 중복이 아니다.

전화번호의 하이픈 유무가 다양하므로 기본 키 비교 전에 숫자만 남기는 정규화 정책이 필요하다. 이 정책을 사용하지 않으면 동일한 실제 번호가 서로 다른 문자열 키로 저장될 수 있다.

### 5.2 `src/main/resources/success_address.csv`

이름과 달리 유효하지 않은 데이터와 중복 데이터가 포함된 검증용 샘플로 보인다.

- 중복 이메일이 존재한다.
- 이메일 형식 오류가 존재한다.
- `010-00-0004`와 같이 허용되지 않은 전화번호 형식이 존재한다.

운영 데이터 파일이 아니라 초기 로딩의 성공/실패 혼합 시나리오를 검증하기 위한 테스트 픽스처로 사용한다. 실제 원본은 프로젝트 루트의 `default_address.csv`로 확정했다.

## 6. 요구사항에서 도출한 도메인 규칙

고객은 다음 네 필드로 구성된다.

| 필드 | 규칙 | 검색 | 정렬 |
|---|---|---|---|
| 전화번호 | 필수, 허용 형식 준수, `010` 시작, 기본 키 | 정확 일치 | 지원 |
| 이메일 | 필수, `아이디@도메인` 형식, 전체 고객에서 유일 | 정확 일치 | 지원 |
| 주소 | 필수, 공백 문자열 불가 | 부분 문자열 포함 검색 | 지원 |
| 이름 | 필수, 공백 문자열 불가 | 부분 문자열 포함 검색 | 지원 |

추가로 필요한 불변 조건은 다음과 같다.

- 전화번호 키와 이메일 유일성은 하나의 쓰기 작업 안에서 원자적으로 확인하고 반영해야 한다.
- 전화번호는 실무에서 일반적으로 기본 키로 선택하지 않는 변경 가능한 자연 키지만, 과제에서 명시한 기본 키이므로 수정 시 변경을 허용한다. 경로의 기존 전화번호로 대상을 식별하고 기존 키 제거와 새 키 추가를 하나의 원자적 작업으로 처리한다.
- 이메일은 유효한 `아이디@도메인` 입력 문자열을 대소문자 변경 없이 저장·응답한다. 중복 검사와 정확 조회는 저장 문자열을 기준으로 대소문자를 구분한다.
- 이메일 형식은 요구사항의 최소 범위에 맞춰 공백과 `@`가 아닌 문자가 `@` 앞뒤에 각각 한 글자 이상 존재하고 `@`가 하나인지를 검증한다. 도메인의 점(`.`) 포함 여부는 요구사항에 없으므로 강제하지 않는다.
- 유효성 검증 규칙은 초기 CSV 로딩과 API 입력에 동일하게 적용하되, 오류 처리 방식은 달라야 한다.
  - 초기 로딩: Warning 로그 후 해당 행만 건너뛰고 계속 진행
  - API: 적절한 4xx 상태 코드와 사용자 메시지 반환
- API 응답에 내부 예외 메시지나 스택 트레이스를 직접 노출하지 않는다.

## 7. 제안하는 목표 아키텍처

과제 규모에서 지나친 프레임워크 추상화는 피하되, HTTP·업무 규칙·메모리·파일 책임은 분리한다.

```text
Controller
  │  요청 검증 및 HTTP 변환
  ▼
Application Service
  │  조회/수정/삭제 유스케이스와 트랜잭션 경계
  ▼
Customer Repository 인터페이스
  │
  ├── InMemoryCustomerRepository
  │     ├── 전화번호 Map
  │     ├── 이메일 보조 Map
  │     └── 읽기/쓰기 동시성 제어
  │
  └── CSV Persistence Adapter
        ├── 시작 시 스트리밍 로딩
        ├── 행 단위 파싱/검증/Warning
        └── 종료 시 백업 → 임시 파일 → 교체
```

예상 패키지는 다음과 같다.

```text
com.hyundai.test.address
├── controller
│   ├── AddressBookController
│   ├── dto
│   └── GlobalExceptionHandler
├── service
│   └── AddressBookService
├── domain
│   ├── Customer
│   ├── CustomerSearchCondition
│   └── 정렬 관련 enum
├── repository
│   ├── CustomerRepository
│   └── InMemoryCustomerRepository
├── persistence
│   ├── CsvCustomerStore
│   └── CustomerDataLifecycle
├── validation
│   └── CustomerValidator
└── exception
```

## 8. 목표 데이터 흐름

### 8.1 시작 시 로딩

1. 외부 CSV 파일을 UTF-8로 연다.
2. 헤더를 검증한다.
3. 행을 하나씩 파싱한다.
4. 필드 형식과 필수값을 검증한다.
5. 전화번호를 비교용 키로 정규화한다.
6. 전화번호와 이메일 중복을 확인한다.
7. 정상 행만 메모리 저장소에 추가한다.
8. 잘못된 행은 행 번호와 사유를 Warning 로그로 남기고 다음 행을 처리한다.
9. 최종 적재/건너뜀 건수를 로그로 남긴다.

전체 파일 문자열 목록과 도메인 목록을 동시에 유지하지 않고 스트리밍 처리하여 중간 메모리 사용을 제한한다.

### 8.2 조회

1. Controller가 필터와 정렬 파라미터를 타입 안전한 검색 조건으로 변환한다.
2. 전화번호 또는 이메일 정확 검색은 보조 인덱스를 통해 평균 O(1)로 후보를 찾는다.
3. 이름 또는 주소 부분 검색은 메모리 스냅샷을 순회하여 O(n)으로 처리한다.
4. 결과를 요청한 필드와 방향으로 정렬한다.
5. 조회 결과를 불변 응답 DTO 목록으로 변환한다.
6. Controller는 조회 결과 및 오류 유형에 대응하는 HTTP 상태 코드와 함께 JSON을 반환한다.

이름/주소의 임의 부분 문자열 검색은 일반적인 정렬 인덱스로 가속하기 어렵다. 과제 범위와 데이터 크기가 정해지지 않은 상황에서 n-gram 같은 고비용 인덱스를 추가하면 쓰기 비용과 메모리 복잡도가 크게 증가한다. 우선 O(n) 스캔을 사용하고, 성능 기준이나 대규모 데이터 요구가 확인될 경우 별도 검색 인덱스를 확장하는 것이 합리적이다.

### 8.3 수정

1. `PUT /api/customers/{phoneNumber}`의 경로 값으로 수정 대상의 기존 전화번호 키를 식별한다.
2. 요청 전체를 검증한다.
3. 쓰기 잠금 안에서 대상 존재 여부, 새 전화번호 충돌, 새 이메일 충돌을 확인한다.
4. 전화번호 Map과 이메일 보조 Map을 함께 갱신한다.
5. 변경 전/후의 불변 스냅샷을 반환한다.

요청 본문에는 변경 후 고객정보 전체를 받는다. 경로 전화번호는 현재 대상을, 본문의 전화번호는 변경 후 키를 의미하므로 전화번호 변경 여부를 모호하지 않게 판단할 수 있다.

### 8.4 삭제

1. 삭제 조건을 검증한다.
2. 전화번호/이메일은 정확 일치, 이름/주소는 포함 검색으로 대상을 결정한다.
3. 쓰기 잠금 안에서 전화번호 Map과 이메일 보조 Map에서 대상들을 함께 제거한다.
4. 삭제된 고객 목록을 반환한다.

이름과 주소는 여러 고객을 찾을 수 있으므로 삭제 응답은 단건이 아니라 목록이어야 일관된다.

### 8.5 정상 종료 시 저장

1. 저장에 사용할 일관된 메모리 스냅샷을 확보한다.
2. 기존 원본 파일을 타임스탬프가 포함된 백업 파일로 복사한다.
3. 같은 디렉터리의 임시 파일에 헤더와 현재 고객 데이터를 UTF-8 CSV로 쓴다.
4. 쓰기가 완료된 뒤 임시 파일을 원본 경로로 교체한다.
5. 가능하면 원자적 이동을 사용하고, 파일 시스템이 지원하지 않으면 안전한 대체 이동을 사용한다.
6. 실패 시 원본과 백업을 보존하고 오류를 로그로 남긴다.

원본에 직접 쓰다가 프로세스가 중단되면 파일이 일부만 기록될 수 있으므로 임시 파일을 거치는 방식이 필요하다.

원본 파일이 없거나 읽기 권한이 없는 경우에는 빈 주소록으로 기동하지 않고 기동을 실패시킨다. 데이터 장애를 정상적인 빈 상태로 숨기고 종료 시 빈 파일로 덮어쓸 위험이 있기 때문이다. 파일을 정상적으로 읽었지만 유효한 데이터가 0건인 경우에는 Warning 및 적재 결과 로그를 남기고 빈 주소록으로 기동한다.

## 9. 동시성 설계 판단

### 9.1 자료구조

제안 자료구조는 다음과 같다.

- 전화번호 기본 저장소: `Map<String, Customer>`
- 이메일 유일성 및 정확 검색 인덱스: `Map<String, String>` (`email -> phoneKey`)
- 저장 객체: 수정 가능한 필드가 없는 불변 객체

`ConcurrentHashMap` 두 개만 사용하면 개별 Map 작업은 안전하지만 두 Map 사이의 원자성은 보장되지 않는다. 따라서 복합 읽기/쓰기 정책이 추가로 필요하다.

### 9.2 잠금

저장소 경계에서 `ReentrantReadWriteLock`을 사용한다.

- 조회와 스냅샷 생성은 읽기 잠금
- 추가, 수정, 삭제, 인덱스 갱신은 쓰기 잠금
- 하나의 쓰기 임계 구역에서 전화번호와 이메일 인덱스를 함께 변경

#### 9.2.1 잠금이 필요한 이유

저장소는 전화번호 기본 Map과 이메일 보조 인덱스라는 두 자료구조를 함께 관리한다. 수정과 삭제는 다음 여러 단계를 하나의 논리적 작업으로 처리해야 한다.

1. 기존 고객과 중복 여부를 확인한다.
2. 전화번호 Map을 변경한다.
3. 이메일 인덱스를 변경한다.
4. 두 자료구조가 같은 고객 집합을 가리키는 상태로 작업을 끝낸다.

각 Map의 개별 연산만 스레드 안전해도 이 단계들 사이에 다른 요청이 개입하면 한쪽 Map만 변경된 중간 상태가 노출될 수 있다. 따라서 자료구조가 아니라 저장소의 복합 연산 전체를 보호하는 잠금이 필요하다.

#### 9.2.2 대안 비교

| 선택지 | 장점 | 단점 | 이 프로젝트에서의 판단 |
|---|---|---|---|
| `synchronized` 또는 단일 `ReentrantLock` | 구현이 가장 단순하고 복합 쓰기의 원자성을 보장하기 쉽다 | 읽기끼리도 직렬화되어 조회 동시성이 불필요하게 제한된다 | 정확하지만 조회가 쓰기보다 많을 주소록 특성에 비해 동시성이 낮아 제외 |
| `ConcurrentHashMap` 두 개 | 개별 Map의 단건 연산과 동시 조회 성능이 좋다 | 두 Map을 함께 갱신하는 수정·삭제의 원자성을 보장하지 않는다. 별도 잠금을 추가하면 구조가 더 복잡해진다 | 단독 사용은 불변 조건을 보장하지 못하므로 제외 |
| `StampedLock` | 낙관적 읽기로 읽기 경합이 매우 높은 환경에서 처리량을 높일 수 있다 | 재진입이 불가능하고 사용법이 복잡하다. 낙관적 읽기 후 검증·재시도 누락 위험이 있으며 전체 스냅샷 조회에서는 이점이 제한적이다 | 측정된 고경합 요구가 없는 과제 규모에서는 복잡성 대비 이점이 작아 제외 |
| 불변 상태 + `AtomicReference` 또는 copy-on-write | 조회에 잠금이 거의 필요 없고 일관된 스냅샷을 쉽게 제공한다 | 쓰기마다 전체 Map을 복사하므로 O(n) 시간과 추가 메모리가 필요하다. 수정·삭제 빈도와 데이터 크기가 커질수록 비용이 급증한다 | 쓰기가 극히 드물고 데이터가 작다는 보장이 없어 제외 |
| `ReentrantReadWriteLock` | 여러 조회가 동시에 진행될 수 있고, 쓰기 잠금 하나로 두 Map의 복합 변경을 원자적으로 보호할 수 있다. 잠금 범위와 테스트가 명확하다 | 읽기 작업이 길면 쓰기가 대기한다. 잘못된 잠금 순서나 잠금 해제 누락에 주의해야 한다. 읽기 경쟁이 낮으면 단일 잠금보다 오버헤드가 있다 | 요구사항의 동시성, 구현 명확성, 읽기 중심 작업 특성 사이의 균형이 가장 적절하여 선정 |

#### 9.2.3 선정 기준

선정 우선순위는 다음과 같다.

1. 전화번호 Map과 이메일 인덱스의 일관성을 모든 쓰기 작업에서 보장할 것
2. 조회 요청끼리는 안전하게 병렬 처리할 수 있을 것
3. 단위 테스트와 동시성 테스트로 동작을 명확히 검증할 수 있을 것
4. 현재 요구 규모에 비해 과도한 메모리 복사나 복잡한 낙관적 제어를 도입하지 않을 것

이 기준에서 `ReentrantReadWriteLock`은 단일 배타 잠금보다 읽기 동시성이 좋고, `ConcurrentHashMap` 단독 사용보다 복합 불변 조건을 명확하게 보장하며, `StampedLock`보다 구현과 검증이 단순하다.

#### 9.2.4 적용 세부 원칙

- 고객 객체는 불변 객체로 만들어 잠금 밖에서 객체 내부 값이 바뀌지 않게 한다.
- 잠금은 저장소 구현 내부에서만 획득하고 서비스나 Controller로 노출하지 않는다.
- 모든 잠금 해제는 `finally`에서 수행한다.
- 정확 조회는 읽기 잠금 안에서 필요한 값만 가져온다.
- 전체/부분 조회는 읽기 잠금 안에서 일관된 고객 목록 스냅샷만 만든 뒤 잠금을 해제하고, 필터링과 정렬은 잠금 밖에서 수행해 쓰기 대기 시간을 줄인다.
- 추가·수정·삭제는 중복 확인부터 두 인덱스 갱신 완료까지 하나의 쓰기 잠금으로 보호한다.
- 잠금 획득 중 파일 I/O, 로그 포맷팅, JSON 변환 같은 느린 작업을 수행하지 않는다.
- 기본적으로 처리량을 우선하는 비공정 모드를 사용하되, 동시성 QA에서 지속적인 쓰기 지연이 관찰되면 공정 모드 전환을 검토한다.

예상 읽기 비율이나 실제 경합이 확인되지 않은 상태이므로 더 복잡한 방식이 항상 빠르다고 가정하지 않는다. 구현 후 동시성 QA 결과를 기준으로 잠금 대기나 쓰기 기아가 확인될 때만 대안을 재검토한다.

## 10. 성능 및 메모리 판단

예상 복잡도는 다음과 같다.

| 작업 | 예상 복잡도 |
|---|---|
| 전화번호 조회 | 평균 O(1) |
| 이메일 조회 | 평균 O(1) |
| 이름/주소 포함 조회 | O(n) |
| 정렬 포함 조회 | O(n log n) |
| 전화번호/이메일 단건 수정 | 평균 O(1) |
| 전화번호/이메일 단건 삭제 | 평균 O(1) |
| 이름/주소 조건 다건 삭제 | O(n) |
| 종료 스냅샷 및 저장 | O(n) |

검색 결과를 만들기 위해 전체 고객 복사본을 상시 유지하지 않고 요청 시 필요한 스냅샷만 만든다. 종료 저장도 파일 출력에 필요한 한 번의 스냅샷만 사용한다.

페이지네이션은 요구사항에 없지만 데이터가 커질 경우 무제한 목록 JSON 응답과 전체 정렬이 병목이 된다. 현재 MVP에서는 API 계약을 임의 확장하지 않고, 문서에 확장 지점으로 남긴다.

## 11. 오류 및 HTTP 응답 설계 방향

추가된 요구사항에 따라 조회 결과와 오류를 단순 JSON 메시지만으로 구분하지 않고 HTTP 상태 코드로 표현해야 한다.

계층별 책임은 다음과 같이 분리한다.

- Service는 HTTP에 의존하지 않고 조회 결과 또는 의미 있는 도메인/애플리케이션 예외를 반환한다.
- Controller와 전역 예외 처리기는 서비스 결과를 HTTP 상태 코드와 응답 DTO로 변환한다.
- 성공 응답도 명시적인 상태 코드를 사용하고, 오류 응답 형식은 전역 예외 처리기로 통일한다.
- 동일한 오류 유형은 어느 API에서 발생하더라도 같은 상태 코드와 오류 코드로 응답한다.

예상 매핑은 다음과 같다.

| 상황 | 상태 코드 |
|---|---|
| 조회 성공 | 200 OK |
| 수정 성공 | 200 OK |
| 삭제 성공 | 200 OK |
| 잘못된 필드 형식, 빈 필수값, 잘못된 정렬 값 | 400 Bad Request |
| 수정/삭제 대상 없음 | 404 Not Found |
| 전화번호 또는 이메일 중복 | 409 Conflict |
| 목록 조회 결과 없음 | 200 OK와 빈 배열 |
| 예상하지 못한 서버 오류 | 500 Internal Server Error |

목록 조회의 0건 결과는 요청 자체가 정상 처리된 빈 컬렉션이므로 `200 OK`와 `[]`를 반환한다. 수정과 삭제처럼 특정 대상을 반드시 찾아야 하는 작업에서 대상이 없으면 `404 Not Found`를 반환한다.

오류 로그에는 예외 스택 트레이스를 기록하되, API에는 오류 코드, 사용자 메시지, 요청 경로, 발생 시각 정도만 반환한다.

## 12. 확정된 정책과 남은 기술 리스크

### 12.1 사용자 답변으로 확정된 정책

1. 조회는 `GET /api/customers`, 수정은 `PUT /api/customers/{phoneNumber}`, 삭제는 `DELETE /api/customers`를 사용한다. 수정 경로의 `phoneNumber`는 현재 저장된 전화번호를 뜻한다.
2. 조회의 복수 필터는 AND로 결합하고 필터가 없으면 전체 조회한다.
3. 삭제 조건은 전화번호, 이메일, 주소, 이름 중 정확히 하나만 허용한다.
4. 이름과 주소의 “양방향 like”는 검색어가 어느 위치에든 포함되는 대소문자 구분 없는 검색으로 처리한다.
5. 이름, 주소, 전화번호, 이메일은 모두 필수값이다.
6. 전화번호는 숫자만으로 동일성을 비교한다. 10자리 번호는 `010-XXX-XXXX`, 11자리 번호는 `010-XXXX-XXXX` 형식으로 저장·응답한다.
7. 이메일은 유효한 `아이디@도메인` 입력 문자열을 그대로 저장·응답하며, 중복 검사와 정확 조회는 대소문자를 구분한다.
8. 수정 대상은 경로의 기존 전화번호로 식별하고 요청 본문의 전화번호 변경을 원자적으로 지원한다.
9. 프로젝트 루트의 `default_address.csv`를 원본으로 사용하고 같은 디렉터리에 타임스탬프 백업을 생성한다.
10. 목록 조회 결과가 0건이면 `200 OK`와 빈 배열을 반환하고 수정·삭제 대상이 없으면 `404 Not Found`를 반환한다.
11. 원본 파일이 없거나 읽을 수 없으면 기동에 실패한다. 파일을 정상적으로 읽은 결과가 0건이면 빈 주소록으로 기동한다.

### 12.2 남은 기술 리스크

1. 비정상 종료는 Java 종료 콜백만으로 저장을 보장할 수 없다. 요구사항은 정상적인 애플리케이션 종료로 한정해야 현실적으로 검증 가능하다.
2. 현재 경로의 한글 정규화 문제 때문에 Windows에서 절대 경로를 클래스패스로 전달하면 Gradle/JUnit 클래스 로딩이 실패한다. 상대 클래스패스로 Gradle Wrapper를 실행하면 컴파일까지 진행되지만 테스트 실행기의 클래스패스에서도 같은 문제가 발생했다. 구현 완료 후 QA 시 영문 경로 복사본 또는 경로 정규화가 안정적인 실행 환경에서 재검증할 필요가 있다.
3. 정렬 파라미터를 생략했을 때의 기본 정렬은 사용자 답변에 따라 전화번호 오름차순으로 확정했다.
4. 원본 파일이 존재하지만 0바이트인 경우에는 사용자 답변에 따라 Warning 로그 후 빈 주소록으로 기동한다. 내용이 있으나 헤더가 잘못된 파일은 구조 오류로 기동 실패 처리한다.

## 14. 기준 빌드 결과

분석 시점의 기준 빌드 결과는 다음과 같다.

- Java 컴파일: 성공
- 테스트 컴파일: 성공
- 테스트 실행: 실패
- 실패 원인: 테스트 코드의 assertion 실패가 아니라, 현재 작업 디렉터리의 한글 정규화가 포함된 절대 클래스패스에서 Gradle 테스트 실행기가 컴파일된 테스트 클래스를 찾지 못함
- 별도 문제: 기본 `gradlew.bat`도 절대 클래스패스를 구성하므로 같은 경로에서 Wrapper 메인 클래스를 찾지 못함

이는 애플리케이션 비즈니스 로직의 기존 실패가 아니라 실행 환경/경로 문제다. 다만 현재 테스트 클래스가 비어 있으므로, 경로 문제를 제외해도 요구사항 검증은 전혀 수행되지 않는 상태다.
