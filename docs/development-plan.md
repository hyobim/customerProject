# 개발 계획 및 승인 요청

## 1. 진행 원칙

- 이 계획은 루트 `Readme.md`, `docs/requirements.md`, `docs/constraints.md`, 기존 코드의 분석 결과를 바탕으로 작성했다.
- 세 명세 문서가 충돌하지 않는 항목은 합집합으로 적용하고, 더 구체적인 조건은 해당 조건을 따른다.
- 사용자 답변으로 정책을 확정했으며, 최종 개발 계획 승인을 받은 뒤 구현을 시작한다.
- 모든 구현은 Java로 작성하고 RDBMS 및 임베디드 DB를 사용하지 않는다.
- 기능 구현과 함께 테스트와 한국어 문서를 갱신한다.
- 각 단계에서 컴파일 및 관련 테스트를 먼저 확인하고, 마지막에 전체 단위 테스트와 통합 테스트를 수행한다.
- 구현 과정에서 계획을 바꿔야 할 중요한 사실이 발견되면 근거를 문서화하고 사용자에게 알린다.

## 2. 단계별 개발 계획

### 2.1 API 계약과 도메인 규칙 확정

- 아래 “승인 전 확인 질문”의 답변을 반영한다.
- 확정된 URL, 요청 파라미터, 요청/응답 JSON, 상태 코드를 API 명세 문서에 기록한다.
- 조회 성공, 조회 결과 없음, 잘못된 검색 조건, 수정/삭제 대상 없음, 중복 충돌, 서버 오류의 HTTP 상태 코드 매핑 표를 확정한다.
- 전화번호 정규화, 이메일 중복 비교, 필수값 정책을 확정한다.

산출물:

- `docs/api-specification.md`
- `docs/implementation-decisions.md` 초기 버전

### 2.2 도메인 모델과 공통 검증 구현

- 불변 고객 모델을 작성한다.
- 조회 필터, 정렬 필드, 정렬 방향을 타입으로 모델링한다.
- 전화번호, 이메일, 필수값 검증을 한 곳에 모은다.
- CSV 로딩과 API 입력이 같은 검증 규칙을 재사용하도록 한다.
- 사용자 입력 오류와 중복/미존재 오류를 구분하는 예외 계층을 작성한다.

검증 항목:

- 허용되는 세 가지 전화번호 형식
- `010` 시작 여부
- 이메일의 `아이디@도메인` 기본 형식: 공백과 `@`가 아닌 문자가 `@` 앞뒤에 각각 한 글자 이상 존재하고 `@`는 하나
- 전화번호 정규화 후 동일 키 충돌
- 이메일 원문 기준 대소문자를 구분한 유일성
- `null`, 빈 문자열, 공백 문자열

### 2.3 메모리 저장소 구현

- 전화번호 기본 Map과 이메일 보조 인덱스를 구현한다.
- `ReentrantReadWriteLock`으로 복합 자료구조의 일관성을 보장한다.
- 읽기 잠금에서는 필요한 값 또는 일관된 목록 스냅샷만 확보하고, 부분 검색과 정렬은 잠금 밖에서 수행한다.
- 쓰기 잠금은 중복 검사부터 전화번호 Map과 이메일 인덱스 갱신 완료까지 유지한다.
- 파일 I/O와 응답 변환은 잠금 임계 구역에 포함하지 않는다.
- 정확 검색과 부분 검색을 분리한다.
- 수정 시 변경 전/후 스냅샷을 반환한다.
- 다건 삭제 시 두 인덱스를 원자적으로 갱신한다.
- 외부에서 내부 컬렉션이나 변경 가능한 객체를 참조할 수 없게 한다.

집중 검증:

- 동시에 같은 전화번호를 추가/수정하는 경우 한 건만 성공
- 동시에 같은 이메일을 사용하는 경우 한 건만 성공
- 전화번호 변경 중 기존 키와 새 키가 동시에 노출되지 않음
- 삭제 후 이메일 인덱스가 남지 않음
- 조회 결과 정렬이 저장소 내부 순서에 의존하지 않음

### 2.4 CSV 파일 영속성 구현

- 프로젝트 외부 파일 시스템의 CSV를 UTF-8로 읽고 쓴다.
- 샘플과 동일한 `주소,연락처,이메일,이름` 헤더 및 순서와 데이터 행의 열 개수를 검증한다.
- 행 단위로 파싱하고 잘못된 행은 Warning 로그 후 건너뛴다.
- 적재 성공/실패 건수를 기록한다.
- CSV 특수문자 처리를 위해 검증된 CSV 라이브러리 사용을 우선 검토한다.
- 종료 시 원본 백업, 임시 파일 쓰기, 원본 교체 순으로 저장한다.
- 파일 경로는 설정값으로 주입하여 테스트에서 임시 디렉터리를 사용할 수 있게 한다.

집중 검증:

- 파일 없음, 읽기 불가, 0바이트 파일, 헤더만 있는 파일, 헤더 오류
- 열 부족/초과
- 잘못된 전화번호/이메일
- 중복 전화번호/이메일
- 한글과 UTF-8
- 필드 안의 쉼표, 따옴표, 줄바꿈
- 일부 행 오류 후 후속 정상 행 적재
- 저장 실패 시 기존 원본 보존
- 백업 파일 내용과 저장된 최종 파일 내용

### 2.5 애플리케이션 서비스 구현

- 조회, 수정, 삭제 유스케이스를 제공한다.
- Controller가 저장소 세부 구현을 알지 않도록 한다.
- HTTP 상태 코드에 직접 의존하지 않고, Controller가 판별할 수 있는 조회 결과와 도메인 예외를 제공한다.
- 다중 조건과 미검색 결과 정책을 확정된 API 계약대로 처리한다.
- 수정 응답에 변경 전/후 데이터를 명확히 분리한다.
- 삭제 응답은 다건 삭제 가능성을 반영한다.

### 2.6 REST Controller 및 예외 응답 구현

- 조회, 수정, 삭제 엔드포인트를 구현한다.
- 요청 DTO와 응답 DTO를 도메인 모델과 분리한다.
- Spring Bean Validation을 사용해 HTTP 경계에서 기본 형식을 검증한다.
- Controller와 전역 예외 처리기에서 조회 결과 및 오류 유형을 200, 400, 404, 409, 500 등 확정된 HTTP 상태 코드로 변환한다.
- 동일한 오류 유형에 대해 API별로 서로 다른 상태 코드가 반환되지 않도록 중앙 매핑 정책을 적용한다.
- 사용자 응답에는 안전한 메시지만 포함하고 서버 로그에는 스택 트레이스를 기록한다.

### 2.7 시작/종료 생명주기 연결

- 애플리케이션 시작 시 CSV를 읽어 저장소를 초기화한다.
- 로딩 중 일부 행 오류는 기동 실패로 전파하지 않는다.
- 원본 파일이 없거나 읽을 수 없으면 애플리케이션 기동을 실패시킨다.
- 파일을 정상적으로 읽었지만 유효 데이터가 0건이면 Warning 및 적재 결과 로그를 남기고 빈 주소록으로 기동한다.
- 정상 종료 콜백에서 백업과 최종 저장을 수행한다.
- 테스트에서 종료 프로세스를 직접 호출해 결과를 검증할 수 있도록 생명주기 로직을 별도 컴포넌트로 둔다.

### 2.8 문서와 README 갱신

- 기존 `Readme.md`의 파일명과 과제 요구사항은 보존하여 구현 결과와 원본 요구사항을 함께 추적할 수 있게 한다.
- 실제 구현 구조와 데이터 흐름을 반영한다.
- 실행 방법과 설정 방법을 작성한다.
- API 요청/응답 예시를 작성한다.
- HTTP 상태 코드 매핑 표를 작성한다.
- 동시성, 자료구조, 검색 복잡도, 파일 안전성의 선택 근거와 한계를 기록한다.
- 비정상 종료 시 저장을 완전히 보장할 수 없는 운영상 한계를 명시한다.

### 3.2 긍정/부정 테스트 균형

- 하나의 테스트에 여러 의미를 과도하게 넣지 않고 실패 원인이 분명하도록 작성한다.

## 4. QA 및 재검증 계획

### 4.1 1차 자동 검증

- `clean test` 전체 실행
- 테스트 리포트 확인
- 성공/실패/스킵 테스트 수 확인
- 테스트 간 임시 파일 및 공유 상태 격리 확인

### 4.2 수동 API QA

- 임시 복제한 CSV로 애플리케이션 실행
- 조회, 정렬, 수정, 삭제의 정상/오류 HTTP 요청 수행
- JSON 구조와 상태 코드 확인
- 조회 결과가 있는 경우와 없는 경우의 상태 코드 및 응답 본문 확인
- 서버 로그에 Warning 및 예외 로그가 의도대로 남는지 확인
- 정상 종료 후 백업 파일과 최종 CSV 비교

### 4.3 성능/동시성 QA

- 충분한 수의 인메모리 고객을 생성하여 정확 검색과 부분 검색의 특성 확인
- 병렬 수정/삭제 요청에서 키와 이메일 인덱스 일관성 확인
- 종료 스냅샷 중 읽기/쓰기 정책 확인
- 무제한 조회 결과가 매우 클 때의 한계를 문서화

### 4.4 재검증

- QA에서 발견한 결함 수정
- 영향받는 단위 테스트 추가
- 전체 테스트 재실행
- 주요 API 수동 회귀 테스트 재수행
- 최종 CSV와 백업 복구 가능성 재확인
- `docs/test-and-qa-report.md`에 명령, 환경, 결과, 알려진 한계를 기록

현재 작업 경로의 한글 정규화로 Gradle/JUnit 클래스 로딩 문제가 있으므로, 최종 검증은 가능하면 영문 경로의 동일 복사본에서도 한 번 더 수행한다. 이는 소스 변경이 아니라 실행 환경 교차 검증으로 취급한다.

## 5. 완료 기준

- 요구사항의 조회, 수정, 삭제가 문서화된 REST API로 동작한다.
- 조회 결과와 오류 유형이 확정된 HTTP 상태 코드 정책에 따라 응답된다.
- 초기 CSV의 잘못된 행은 Warning 후 건너뛰고 정상 행은 계속 적재된다.
- 전화번호 키와 이메일 유일성이 단일 요청 및 동시 요청에서 유지된다.
- 정상 종료 시 원본 백업과 메모리 데이터 저장이 검증된다.
- 사용자 오류는 HTTP 상태와 안전한 JSON 메시지로 반환된다.
- 모든 프로덕션 클래스가 테스트 대상표에 포함되고, 각 클래스의 계약이 단위 테스트 또는 가장 가까운 계층 테스트에서 검증된다. Controller는 MockMvc로 검증하며 단위 테스트 이후 통합 테스트를 수행한다.
- 긍정/부정 테스트 수가 동일하고 QA 문서에서 확인 가능하다.
- 전체 테스트와 재검증 결과가 문서화된다.
- README와 docs가 실제 구현과 일치하며 모두 한국어로 작성된다.

## 6. 승인 전 확인 질문

아래 항목은 요구사항만으로 단일한 결론을 내리기 어렵고 API 및 저장 설계에 직접 영향을 준다.

1. API 계약을 다음 제안으로 확정해도 되는가?
   - 조회: `GET /api/customers`
   - 수정: `PUT /api/customers/{phoneNumber}`
   - 삭제: `DELETE /api/customers`
